# 架构检查清单

> 用于检查当前仓库的文档、Prompt 契约、插件链路和前端工作流是否仍然对齐“两条主能力”这一目标。

## 1. 项目定位

- [ ] 文档是否仍然围绕 Figma + React 连调
- [ ] 是否清楚表达“Codex / Claude -> Figma”和“Figma -> AI -> React”两条链路
- [ ] 是否把“计划”误写成“已实现”
- [ ] 是否引入了与仓库现状无关的产品语义

## 2. 设计输入

- [ ] 设计来源是否明确来自 Figma 或人工评审结论
- [ ] 组件、页面、状态命名是否一致
- [ ] 是否区分了设计事实与实现推断

## 3. React 映射

- [ ] 组件边界是否清楚
- [ ] 状态、事件、Props 映射是否可执行
- [ ] 是否存在只描述视觉、不描述交互契约的空缺

## 4. AI Runtime 契约

- [ ] `Context Pack` 输入是否有明确字段含义
- [ ] 输出是否仍为结构化 JSON，而不是自由文本
- [ ] action prompt 之间是否出现职责重叠或冲突
- [ ] 失败时是否要求提问而不是瞎猜
- [ ] 是否清楚区分 AI 使用方式与仓库当前真实实现

## 5. 文档系统一致性

- [ ] `README.md`、`AGENT.md`、架构文档、ADR 之间没有互相打架
- [ ] 文件名、文内引用、目录说明一致
- [ ] 没有残留旧项目名或旧产品语义
- [ ] Plugin API、bridge、MCP 的描述没有混淆

## 6. 实施前检查

- [ ] 本次准备实现的内容是否已有文档约束
- [ ] 未确认的设计问题是否先记录为假设或待决事项
- [ ] 是否明确了最小验收范围

## 7. 实施后检查

- [ ] 文档是否随着实现更新
- [ ] Prompt 契约是否仍能支撑当前流程
- [ ] 新增术语是否已定义
- [ ] 没有因为实现方便而破坏原有设计边界
- [ ] `workspace`、`plugin`、`bridge` 是否仍保持职责隔离

## 8. 常见违规示例

- 把“可能会做的 React 目录结构”写成当前仓库事实
- 在 Prompt 中要求 AI 访问并不存在的字段
- 设计文档里只写视觉描述，不写状态和交互
- ADR 已经变更，但 README 和架构文档仍停留在旧口径
- 把 MCP 写成当前仓库已经替代 Plugin API 的执行链
