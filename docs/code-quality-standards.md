# 代码与文档质量标准

本文档定义 AutoDesign 当前阶段的质量门槛。仓库现在同时包含 Markdown、Vite + React 工作台、本地 Node API 和独立 Figma 插件，因此文档质量、Prompt 契约质量和代码质量都要一起看。

## 1. Markdown 文档质量

### 1.1 事实准确

- 当前仓库中不存在的文件、目录、脚本、页面，不得写成已存在
- 计划中的实现必须标注为“计划”“建议”或“目标态”
- 文档之间对同一事实的描述不能互相冲突

### 1.2 结构清晰

- 顶层标题能直接说明文档作用
- 标题层级连续，不随意跳级
- 长文档优先拆成短段落和清晰列表

### 1.3 命名统一

- 项目名统一为 `AutoDesign`
- 协作主文档统一引用 `AGENT.md`
- 术语首次出现时给出中文解释，字段名保留英文

### 1.4 引用可追踪

- 文内提到的关键文档应能在仓库中找到
- 文件重命名后必须同步更新引用
- 删除旧文档语义时，不能留下孤立说明

## 2. Prompt 与契约质量

### 2.1 输入边界清楚

- action 必须说明依赖哪些 `Context Pack` 字段
- 缺少关键字段时，应要求提问，不允许默认猜测
- 输入语义需要和架构文档一致

### 2.2 输出可消费

- 输出格式必须明确，例如 JSON only
- 需要说明稳定 ID、顺序约束、失败字段或 questions 字段
- 不允许把本应结构化输出的结果写成自由散文

### 2.3 职责单一

- 每个 action 只解决一种明确任务
- 不把摘要、重组、分支规划塞进同一个 Prompt
- action 之间的术语和字段要统一

## 3. React Workspace / Node API / Figma Plugin 代码质量

当前仓库建议使用以下门槛：

- 组件职责清晰，避免单文件承担页面、状态、数据映射和样式四类职责
- 交互状态与设计状态表保持一致
- 设计 token、组件 props、状态枚举优先结构化，不靠隐式约定
- 示例和实验代码要标注用途，避免与正式实现混淆
- 本地 API 的输入输出要稳定，避免前端偷偷依赖未声明字段
- 文件型持久化逻辑与 UI 逻辑分离，避免组件直接操作文件系统
- Figma 插件的写操作必须可结构化描述，避免把关键行为藏进难以审计的 UI 分支

## 4. 评审时重点关注什么

- 文档是否比事实更“完整”
- Prompt 是否真的能被调用方消费
- 新增术语是否解释清楚
- 设计和实现的边界是否被破坏

## 5. 推荐检查方式

```bash
rg -n "AutoDesign|autodesign|AutoDesign_Project_Map" -g '*.md'
rg -n "AGENT\\.md|AutoDesign_Project_Map\\.md" -g '*.md'
find . -type f -name '*.md' | sort
```

当前至少应补充 `npm run build` 作为最小代码验证。
