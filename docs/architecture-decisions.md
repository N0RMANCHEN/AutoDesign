# 架构决策记录（ADR）

> 本文记录 AutoDesign 的关键方向性决策。  
> 当项目定位、Prompt 契约或实施路径发生明显变化时，应补充新的 ADR。

## ADR-001：仓库采用 docs-first 方式推进

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

在 React 原型和联调工具落地前，先以 Markdown 文档沉淀项目目标、架构边界、Prompt 契约和检查规则。

### 决策理由

- 当前最大的不确定性在方案而不是代码
- 先把接口和边界讲清楚，能减少后续返工
- 文档更适合沉淀 Figma 到 React 的跨角色共识

### 替代方案

- 直接写 React 原型，再事后补文档
- 先做 Figma 插件，再回头定义联调规则

### 影响范围

- `README.md`、`AGENT.md`、`docs/architecture*.md` 成为当前主交付
- 文档质量和引用一致性变成主要验证项

## ADR-002：Figma 作为设计输入源，React 作为验证层

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

设计结构、组件命名、状态和交互范围优先从 Figma 或设计评审结论获取；React 原型只承担验证和映射职责。

### 决策理由

- 能避免实现反向定义设计
- 便于区分“设计已确认”与“实现待验证”
- 有利于后续联调时判断差异来自设计还是来自实现

### 替代方案

- 由 React 原型反推设计结构
- 设计和实现混在同一份说明中，不做边界区分

### 影响范围

- 架构文档必须区分设计真相与实现真相
- Prompt 上下文中应显式标注设计信息来源

## ADR-003：AI Runtime 文档保留为接口规范

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

保留 `docs/ai/runtime/` 目录作为 AI Runtime 契约层，而不是删成概念说明。

### 决策理由

- 后续无论是否接入真实模型，输入输出接口都需要先定义
- Prompt、Context Pack、JSON 输出本身就是联调流程的一部分
- 及早收敛 action 边界，可以减少未来实现时的返工

### 替代方案

- 只保留一份概念文档，不写具体 action
- 等实现 AI 面板时再补所有 Prompt

### 影响范围

- 运行时文档修改需要视为接口变更
- action 文档之间必须保持职责清晰

## ADR-004：文档语言以中文为主

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

仓库文档统一使用中文为主，必要术语保留英文。

### 决策理由

- 当前协作语境以中文为主
- 架构、流程、Prompt 说明需要提高直接可读性
- 保留关键英文术语即可满足技术表达需要

### 替代方案

- 全英文文档
- 中英双份并行维护

### 影响范围

- 新文档默认中文标题和正文
- Prompt 文档中的字段名保留英文，解释使用中文

## ADR-005：去除旧项目语义并统一命名

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

将旧的项目命名统一替换为 `AutoDesign`，包括入口文件、项目地图、插件目录与用户可见品牌名。

### 决策理由

- 旧命名会误导协作者理解项目范围
- 文档内容与仓库目标已经不一致
- 保持路径、标题和口径一致，有利于后续继续扩展

### 替代方案

- 只改正文，不改文件名
- 保留双命名并行

### 影响范围

- `docs/AutoDesign_Project_Map.md` 作为统一仓库地图
- 插件目录改为 `plugins/autodesign/` 与 `plugins/autodesign-smoke/`
- 所有 Markdown 引用都需要同步修复

## ADR-006：使用 Next.js App Router + 本地 JSON 后端落 MVP

**日期**：2026-03-22  
**状态**：已替代

### 决策内容

项目首个可运行版本采用 Next.js App Router，实现前端工作台与 Node.js route handlers，并使用本地 JSON 文件作为持久化层。

### 决策理由

- 可以在一个仓库内同时交付前端页面和后端 API
- 便于快速测试 Figma 风格输入、组件映射和 Runtime action
- 在未确定数据库和部署架构前，本地 JSON 足够支持 MVP 验证

### 替代方案

- 先拆成独立前端和后端仓库
- 直接接数据库或云服务
- 只做前端静态原型，不落真实 API

### 影响范围

- 这是一个历史阶段性方案，已不再代表当前实现

## ADR-007：重构为 Vite Workspace + 独立 Node API + 独立 Figma Plugin

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

不再使用 Next.js App Router 作为单体 MVP，而是将仓库重构为三个清晰部分：

- `src/`：Vite + React 工作台
- `server/`：独立本地 Node API
- `plugins/autodesign/`：独立 Figma 插件执行器

并新增 `shared/` 作为共享领域模型与命令契约层。

### 决策理由

- 用户明确要求采用 Vite + React，而不是继续保留 Next.js
- 仓库现在要同时承载 Figma-to-React 和 Codex-to-Figma，两条链路需要明确隔离
- 工作台负责上下文与实现映射，插件负责设计写操作，这两个运行时不适合继续放在同一框架抽象下
- 共享类型和命令协议可以保留一致性，同时减少两个系统之间的耦合

### 替代方案

- 继续使用 Next.js，并在同一应用里模拟插件逻辑
- 把插件能力写成工作台页面，而不是独立 Figma 插件
- 拆成多个仓库，而不是保留单仓共享模型

### 影响范围

- `README.md` 与架构文档需要从单体 MVP 口径改为双系统口径
- 目录结构调整为 `src/`、`server/`、`shared/`、`plugins/`
- 构建与开发脚本从 Next.js 切换为 Vite、TypeScript 和本地 Node 服务器
- 插件写操作必须经过共享命令协议，而不是由工作台直接执行

## ADR-008：Figma 主执行链固定为 Plugin API + 本地 Bridge

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

当前仓库中，AI 对 Figma 的主执行链固定为：

- Figma Plugin API
- 本地 `localhost` bridge
- 本地命令队列与结果回传

MCP 不作为当前仓库的主执行链。

### 决策理由

- 仓库已经有本地插件执行器，可以直接读写 Figma 文件
- Plugin API 更适合当前的私有开发插件形态
- MCP 的 seat、计划与配额规则独立，不能混同为当前实现事实

### 替代方案

- 直接把 MCP 作为当前主执行面
- 同时并列维护两套独立执行链

### 影响范围

- README 和架构文档必须明确区分 Plugin API 与 MCP
- “当前仓库如何改 Figma” 的答案统一为插件执行器
- MCP 只允许被写作未来补充链路或读取链路

## ADR-009：插件能力按 Capability Registry 扩展

**日期**：2026-03-22  
**状态**：已采纳

### 决策内容

插件后续新增能力时，统一通过 capability registry 注册，不再继续扩张单个 `main.ts` 的命令分支。

### 决策理由

- 未来会逐步接入更多 Figma 能力入口
- 需要稳定的能力声明、输入 payload 和结构化结果
- 这样 Codex / Claude / bridge / 插件运行时可以共享同一份能力定义

### 替代方案

- 继续用字符串命令 + 大型 `switch`
- 在 UI 里直接塞越来越多的人工按钮

### 影响范围

- `shared/plugin-capabilities.ts` 成为能力注册入口
- 插件运行时按 capability 组织
- CLI 与 bridge 需要读取 session capabilities 再发命令
